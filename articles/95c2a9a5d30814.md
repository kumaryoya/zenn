## はじめに

AWS のマネコンにログインする際に MFA 認証を導入するのは一般的です。

しかし、セッションハイジャックなどの被害に遭った際のリスクを減らすために、IP 制限を併用することでさらにセキュリティを強化できます。

本記事では、AWS のマネコンでの操作に IP 制限をかける方法について解説します。

少しでも参考になれば幸いです。

セッションハイジャックとは、ユーザーのセッションを乗っ取る攻撃手法です。攻撃者がセッション ID を盗み取り、そのセッションを利用して不正にアクセスします。
:::

- マネコンの操作を特定の IP アドレスに制限したい方
- IAM ポリシーの条件キーに興味がある方
:::

## aws:SourceIp とは

**`aws:SourceIp`** は、IAM ポリシーの **条件（Condition）** で使用できるグローバル条件キーの 1 つです。

リクエスト元の IP アドレスに基づいてアクセスを制御できます。

https://docs.aws.amazon.com/ja_jp/IAM/latest/UserGuide/reference_policies_condition-keys.html#condition-keys-sourceip

:::message
`aws:SourceIp` は **パブリック IP アドレス**を対象とします。VPC 内部のプライベート IP で制限したい場合は、**VPC エンドポイントポリシー**や `aws:VpcSourceIp` を使用する必要があります。
:::

## IP 制限の仕組み

今回の方法は、**IAM ポリシーに「特定の IP アドレス以外からのリクエストを拒否（Deny）する」** というルールを設定するアプローチです。

ポイントは以下の通りです。

- **Allow（許可）ではなく Deny（拒否）で制御する**
- 明示的な Deny は、どの Allow よりも優先される（[明示的な拒否](https://docs.aws.amazon.com/ja_jp/IAM/latest/UserGuide/reference_policies_evaluation-logic.html)）
- `NotIpAddress` 条件を使い、「指定した IP アドレス**以外**」を対象にする

これにより、たとえ `AdministratorAccess` などの強力な権限を持っていても、許可された IP アドレス以外からはリソースの操作ができなくなります。

## 使用するポリシー（共通）

どちらのパターンでも、使用する IAM ポリシーの内容は同じです。

```json:IP 制限ポリシー（JSON）
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyAccessFromUnauthorizedIPs",
            "Effect": "Deny",
            "Action": "*",
            "Resource": "*",
            "Condition": {
                "NotIpAddress": {
                    "aws:SourceIp": [
                        "203.0.113.0/24",
                        "198.51.100.10/32"
                    ]
                },
                "Bool": {
                    "aws:ViaAWSService": "false"
                }
            }
        }
    ]
}
```

:::message
**ポリシーの解説**
- `"Effect": "Deny"` → リクエストを拒否する
- `"Action": "*"` → すべての操作を対象にする
- `"Resource": "*"` → すべてのリソースを対象にする
- `"NotIpAddress"` → 指定した IP アドレス**以外**の場合に条件が一致する
- `"aws:SourceIp"` → 許可する IP アドレスを CIDR 形式で指定する
- `"aws:ViaAWSService": "false"` → AWS サービスが内部的に行うリクエストを制限対象から除外する（後述）
:::

`aws:SourceIp` の部分は、**許可したい IP アドレスに書き換えてください**。

- `/32` → 単一の IP アドレスを指定
- `/24` → サブネット範囲で指定

### aws:ViaAWSService について

上記のポリシーでは `"aws:ViaAWSService": "false"` という条件も追加しています。

これは非常に重要な設定です。AWS のサービスの中には、ユーザーのリクエストを受けて**内部的に別の AWS サービスを呼び出す**ものがあります（例：CloudFormation がリソースを作成する場合など）。

この内部呼び出しの際、リクエスト元 IP はユーザーの IP ではなく **AWS 内部の IP** になります。`aws:ViaAWSService` を指定しないと、こうした内部呼び出しまで拒否されてしまい、正常にサービスが動作しなくなる可能性があります。

https://docs.aws.amazon.com/ja_jp/IAM/latest/UserGuide/reference_policies_condition-keys.html#condition-keys-viaawsservice

## パターン 1：IAM ユーザーの場合

**IAM ユーザー**を使って AWS を利用している場合の設定手順です。

### 1. IAM ポリシーを作成する

**IAM** のコンソールから **ポリシー** → **ポリシーの作成** を選択します。

**JSON** タブを選択し、上記のポリシーを入力します。適切な名前（例：`DenyAccessFromUnauthorizedIPs`）を付けて保存します。

### 2. IAM ユーザー / グループにポリシーをアタッチする

IP 制限を適用したい **IAM ユーザー**または **IAM グループ**にポリシーをアタッチします。

:::message alert
**注意**: ポリシーをアタッチした瞬間から IP 制限が有効になります。自分自身の IP アドレスが許可リストに含まれていることを必ず確認してください。万が一ロックアウトされた場合は、**ルートユーザー**でサインインしてポリシーを修正する必要があります。
:::

#### IAM ユーザーにアタッチする場合

**IAM** → **ユーザー** → 対象のユーザーを選択 → **許可を追加** → **ポリシーを直接アタッチする** から、作成したポリシーを選択してアタッチします。

#### IAM グループにアタッチする場合

複数のユーザーに一括で適用したい場合は、**IAM グループ**にアタッチするのがおすすめです。

**IAM** → **ユーザーグループ** → 対象のグループを選択 → **許可** タブ → **ポリシーをアタッチ** から、作成したポリシーを選択してアタッチします。

### 3. 動作確認

ポリシーをアタッチしたら、以下の観点で動作確認を行います。

1. **許可した IP アドレスから**マネコンにアクセスし、リソースの操作ができることを確認する
2. **許可していない IP アドレスから**マネコンにアクセスし、リソースの操作が拒否されることを確認する

:::message
許可していない IP からアクセスした場合、マネコンにはサインインできますが、各サービスの画面で **「You are not authorized to perform this operation.」** などのエラーが表示されるようになります。

これは、**サインイン自体は IAM ポリシーではなく認証（Authentication）の仕組みで行われる**ためです。今回の設定は認可（Authorization）レベルでの制限になります。
:::

## パターン 2：IAM Identity Center（旧 AWS SSO）の場合

**IAM Identity Center** を使って AWS を利用している場合の設定手順です。

IAM Identity Center では、ユーザーに直接 IAM ポリシーをアタッチするのではなく、**アクセス許可セット（Permission Set）** に IP 制限ポリシーを設定します。

### 1. アクセス許可セットにインラインポリシーを追加する

**IAM Identity Center** のコンソールから **アクセス許可セット** を選択し、IP 制限を適用したいアクセス許可セットを開きます。

**インラインポリシー** タブを選択し、上記の IP 制限ポリシーを追加します。

:::message
**アクセス許可セットにポリシーを設定する方法は 2 つあります**
- **インラインポリシー** → アクセス許可セットに直接ポリシーの JSON を記述する
- **カスタマー管理ポリシー** → 事前に IAM ポリシーを作成しておき、そのポリシー名を指定する

IP 制限のように全アクセス許可セットに共通で適用したいポリシーは、**カスタマー管理ポリシー**として作成しておくと、複数のアクセス許可セットで再利用しやすくなります。ただし、カスタマー管理ポリシーを使う場合は、**対象のすべての AWS アカウントに同名の IAM ポリシーが存在する必要がある**点に注意してください。
:::

### 2. アクセス許可セットの変更を反映する

ポリシーを追加したら、アクセス許可セットの変更を**プロビジョニング（反映）** します。

IAM Identity Center のコンソールで、対象のアクセス許可セットの **AWS アカウント** タブから、変更を反映したいアカウントを選択して **再プロビジョニング** を実行します。

:::message alert
**注意**: 再プロビジョニングを実行すると、そのアクセス許可セットを利用しているすべてのユーザーに IP 制限が即座に適用されます。自分自身の IP アドレスが許可リストに含まれていることを必ず確認してください。
:::

### 3. 動作確認

IAM ユーザーの場合と同様に、許可した IP / 許可していない IP の両方からアクセスして動作を確認します。

:::message
IAM Identity Center 経由でサインインした場合も、許可していない IP からのアクセスでは各サービスの画面でエラーが表示されます。**AWS アクセスポータル（SSO ポータル）へのサインイン自体は IP 制限の対象外**です。
:::

## 応用：特定の操作のみ IP 制限をかける

上記の例では `"Action": "*"` ですべての操作を対象にしましたが、特定の操作のみ制限することも可能です。

例えば、**読み取り操作は許可するが、書き込み操作のみ IP 制限をかける**場合は、以下のように設定できます。

```json:書き込み操作のみ IP 制限をかける例
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyWriteAccessFromUnauthorizedIPs",
            "Effect": "Deny",
            "Action": [
                "ec2:RunInstances",
                "ec2:TerminateInstances",
                "s3:PutObject",
                "s3:DeleteObject",
                "rds:CreateDBInstance",
                "rds:DeleteDBInstance"
            ],
            "Resource": "*",
            "Condition": {
                "NotIpAddress": {
                    "aws:SourceIp": [
                        "203.0.113.0/24"
                    ]
                },
                "Bool": {
                    "aws:ViaAWSService": "false"
                }
            }
        }
    ]
}
```

このように、`Action` に制限したい操作を具体的に指定することで、柔軟な制御が可能です。

## SCP（サービスコントロールポリシー）で組織全体に適用する

**AWS Organizations** を利用している場合は、**SCP（サービスコントロールポリシー）** を使って組織全体や特定の OU（組織単位）に対して IP 制限を適用することもできます。この方法は **IAM ユーザー・IAM Identity Center のどちらにも有効**です。

```json:SCP の例
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Sid": "DenyAccessFromUnauthorizedIPs",
            "Effect": "Deny",
            "Action": "*",
            "Resource": "*",
            "Condition": {
                "NotIpAddress": {
                    "aws:SourceIp": [
                        "203.0.113.0/24"
                    ]
                },
                "Bool": {
                    "aws:ViaAWSService": "false"
                }
            }
        }
    ]
}
```

SCP は IAM ポリシーとは異なり、**対象アカウントのルートユーザーにも適用される**ため、より強力な制限が可能です。ただし、SCP の設定を誤ると組織全体がロックアウトされるリスクがあるため、慎重に設定してください。

https://docs.aws.amazon.com/ja_jp/organizations/latest/userguide/orgs_manage_policies_scps.html

## まとめ

今回は、**AWS マネジメントコンソールでの操作に IP 制限をかける方法**について紹介しました。

- IAM ポリシーの `aws:SourceIp` 条件キーを使うことで、特定の IP アドレスからのみ操作を許可できる
- `Deny` ポリシーと `NotIpAddress` 条件を組み合わせるのがポイント
- `aws:ViaAWSService` を設定して、AWS 内部の呼び出しが拒否されないようにする
- **IAM ユーザー**の場合はポリシーを直接アタッチ、**IAM Identity Center** の場合はアクセス許可セットに設定する
- 組織全体に適用したい場合は SCP の利用も検討する

IP 制限は比較的シンプルに設定できるセキュリティ対策ですが、**設定ミスによるロックアウトのリスク**もあるため、必ず検証環境で十分にテストしてから本番環境に適用することをおすすめします。
