## はじめに

Claude Code や GitHub Copilot など、AI を使ってコードを書くことが当たり前の時代になってきました。

私自身も、日々の開発はほぼすべて Claude Code に任せており、AI なしでコードを書く日はほとんどありません。

そうした開発の日々を送る中で、「AI にコードを書いてもらう時代に、技術記事を書く意味はあるのだろうか？」と考えるようになりました。

実際、公式ドキュメントの要約程度であれば、AI に頼めば数秒で出てきます。

それでも記事を書き続けてきた中で、「やっぱり書いて良かった」と感じる瞬間が何度もあります。

エンジニア未経験のころは Qiita に、エンジニアになってからは Zenn に記事を書いてきました。

その経験を踏まえ、「AI 時代に記事を書いて良かったこと」を整理してみます。

## 結論

AI 時代に記事を書いて良かったことは、以下の 3 つです。

1. **記事を書くと、AI に任せただけでは残らない知識が定着する**
2. **「やってみた」記事こそ、AI 時代の差別化になる**
3. **記事を書くことで、AI への指示力が底上げされていく**

![](https://static.zenn.studio/user-upload/76733fae84d7-20260509.png =300x)

## ① 記事を書くと、AI に任せただけでは残らない知識が定着する

### AI に書かせた日々で起きていたこと

Claude Code に「この機能を実装して」と依頼すると、すぐにコードを書いてくれます。

また、レビュー、テスト、マージといった一連の流れもスムーズに進みます。

しかし、「このコードは何をしているのか」「この SDK はどんな仕組みなのか」といった、中身を理解するプロセスは、AI に任せているだけでは省略されていました。

AI に任せると、手を動かさずに済む代わりに、頭も動かさなくなっていました。

### 記事を書くと、何が起きるか

そこで、AI に任せて実装した領域でも、改めて記事として書き直してみることにしました。

書き始めると、必ずどこかで手が止まります。

- 「ここの仕組みって、結局なんでこうなってるんだっけ？」
- 「この用語、自分は本当に分かって使ってるのか？」

止まった場所が、自分が分かっていなかった場所です。

そこで公式ドキュメントを読み直したり、コードを追い直したりすることで、初めて頭に入ってきます。

書く＝コードや知識を自分の言葉に翻訳する作業であり、AI に任せても省略できないプロセスだと感じています。

## ② 「やってみた」記事こそ、AI 時代の差別化になる

### AI が書ける記事と、AI には書けない記事

AI に「Claude Code とは？」と聞けば、それらしい入門記事は数秒で出てきます。

公式ドキュメントの要約・基本概念の説明・サンプルコードの提示——このあたりは AI が一瞬で再合成してくれる時代になりました。

「自分が同じことを書く意味はあるのか」と感じることも何度もありました。

しかし、しばらく書き続けてきて気づいたのは、**AI が書けるのは「すでに世にある情報の再合成」だけ**だということです。

逆に、

- 実プロジェクトに導入してハマった点
- チームで運用して見えてきた課題
- 検証してみて意外だった結果
- 「やってみた」「やめた」のリアルな判断

このあたりは、**実際にやった人にしか書けない一次情報**です。

### 振り返ると、伸びた記事はだいたい「やってみた」型だった

これまでの自分の記事を振り返ってみると、反応をもらえた記事はたいてい「やってみた」「検証してみた」「劇的に変わった」系でした。

- [Claude Code の `CLAUDE.md` / `skills/` / `agents/` を整備して開発体験が劇的に変わった話](https://zenn.dev/linkedge/articles/27c38cdd9bedc6)
- [GitHub Copilot × Devin で開発をどこまで自動化できたかをまとめた話](https://zenn.dev/linkedge/articles/f04219a0c2f2ad)
- [Knowledge Bases for Amazon Bedrock に矛盾したドキュメントを同期するとどうなるか検証してみた話](https://zenn.dev/linkedge/articles/edacf531030533)

これらに共通しているのは、**「公式ドキュメントには載っていないが、実際にやってみたら分かった」運用知**を書いていることです。

AI が「正解の要約」を出せる時代だからこそ、人間が書く価値は **「試した結果」** にシフトしているのだと感じています。

## ③ 記事を書くことで、AI への指示力が底上げされていく

### 記事を公開する以上、間違ったことは書けない

最初は意外に感じたのですが、**記事を書く習慣はそのまま AI への指示力**につながっています。

記事を公開する以上、適当な用語や曖昧な表現は使えません。

- 「Active Record」と「ActiveRecord」、正しい表記はどっちか
- そのライブラリの正式名称は何か
- 公式ドキュメントには何と書かれているか

——こうしたことを毎回確認する習慣が自然とつきました。

### この習慣が、そのまま AI への指示精度になる

そして、この習慣が AI に指示するときに効いています。

AI は、雑な用語や曖昧な指示を投げると、雑な出力で返してきます。

逆に、**正式名称・前提・制約を正確に渡せば**、出力の精度は明らかに変わります。

たとえば過去に書いた [Claude Code 育成記事](https://zenn.dev/linkedge/articles/27c38cdd9bedc6) で整備した `CLAUDE.md` や `skills/` は、よく考えると **「自分の頭の中にあるプロジェクト前提を、AI に渡せる文章に翻訳する作業」** でした。

これは、記事を書く作業と本質的に同じです。

**記事執筆 = プロンプトエンジニアリングの素振り**——書き続けることで、AI に渡せる文章の解像度が上がっていきます。

## まとめ

AI 時代に記事を書いて良かったことは、以下の 3 つでした。

1. **記事を書くと、AI に任せただけでは残らない知識が定着する**（自分のインプット）
2. **「やってみた」記事こそ、AI 時代の差別化になる**（発信物の市場価値）
3. **記事を書くことで、AI への指示力が底上げされていく**（AI 活用への還流）

並べてみると、書くことは自分の中で完結するのではなく、

> **書く → 自分に入る → 一次情報として外に出る → AI 活用力に還流する**

という循環になっていることに気づきます。

AI が書ける時代だからこそ、人間が書く意味は減るのではなく、**むしろ増えている**——少なくとも自分はそう感じるようになりました。

「AI に任せて開発したけれど、頭に何も残っていない気がする」——もしそんな感覚があれば、まずは 1 本書いてみることをおすすめしたいです。

書くことは、AI 時代の最強の AI 活用法であり、最強の自己投資だと思っています。
