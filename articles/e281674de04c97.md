## はじめに

先日、個人開発で運用しているアプリの動作が遅くなる障害が発生しました。

アプリにアクセスしても、なかなか画面が表示されない状況でした。

調査の結果、原因はアプリのコードやアーキテクチャの致命的なミスではなく、単純な RDS のスペック不足でした。

本記事では、今回の障害対応を通じて、改めて RDS について学んだことを整理してみます。

少しでも皆様の参考になりますと幸いです。

![](https://static.zenn.studio/user-upload/1a46fa2f8afb-20260628.png =300x)

## 前提

本題に入る前に、今回のアプリの構成を簡単に共有しておきます。

- **アプリケーション**：Ruby on Rails
- **実行環境**：Amazon ECS（Fargate）
- **データベース**：Amazon RDS for MySQL

## 調査したこと

何が原因でアプリの動作が遅くなっているのかを調査するため、まずは AWS のマネジメントコンソールで CloudWatch のメトリクスを確認しました。

まずはアプリの実行環境である Fargate、次にデータベースの RDS という順で確認しました。

### Fargate

#### CPU

まずは Fargate の CPU 使用率を確認しましたが、一時的に上がっていることはあっても、基本的には低いまま安定していました。

リクエストを捌けずに CPU が振り切れている、という状況ではなさそうです。

![](https://static.zenn.studio/user-upload/64549816e752-20260628.jpg =1000x)

#### メモリ

メモリ使用率も同様に、特に高騰している様子はありませんでした。

![](https://static.zenn.studio/user-upload/6f252e5fe2b5-20260628.jpg =1000x)

### RDS

#### CPU

次に RDS の CPUUtilization を確認しましたが、こちらも一時的に上がっていることはあっても、基本的には低いまま安定していました。

![](https://static.zenn.studio/user-upload/e84188f0b204-20260628.jpg =1000x)

#### メモリ

FreeableMemory を確認したところ、空きメモリが常時 100MB〜150MB くらいしかなく、低い時では 50MB を切っていました。

![](https://static.zenn.studio/user-upload/3b6a45641498-20260628.jpg =1000x)

#### ReadIOPS / WriteIOPS / TotalIOPS

CPU が高騰していないのにアプリが遅い、ということは、処理そのものが重いのではなく、ディスク I/O の待ちで詰まっている可能性が高いと考え、ディスクの読み書き回数を確認しました。

まずは読み込み回数（ReadIOPS）です。

![](https://static.zenn.studio/user-upload/edc43b169627-20260628.jpg =1000x)

続いて書き込み回数（WriteIOPS）です。

![](https://static.zenn.studio/user-upload/41d4141332ee-20260628.jpg =1000x)

最後に合計回数（TotalIOPS）です。

![](https://static.zenn.studio/user-upload/6f77ad910838-20260628.jpg =1000x)

ここで気になったのが、TotalIOPS が 3,000 近くまで跳ね上がっている時間帯が散見されることです。

読み込み・書き込みともにそれなりの回数が出ており、ディスクへのアクセスがかなりの高負荷になっていることが分かります。

#### DiskQueueDepth ( Maximum )

では、そのアクセスをディスクは捌ききれているのかを確認するため、処理待ちの I/O がどれくらい溜まっているかを表す DiskQueueDepth を確認しました。

![](https://static.zenn.studio/user-upload/054449a32700-20260628.jpg =1000x)

DiskQueueDepth も上昇していました。

これは、ディスクに届いた読み書きのリクエストが処理しきれず、行列を作って待たされている状態です。

アプリの動作が遅かったのは、まさにこの I/O の詰まりが直接の原因だと考えられます。

#### BurstBalance

最後に BurstBalance を確認しました。

使用している RDS のストレージタイプは「汎用 SSD (gp2)」です。

gp2 には「ベースライン性能」という上限があり、容量から計算される一定の IOPS を超えるアクセスは「バーストI/Oクレジット」を消費して一時的に凌ぐ仕組みになっています。

使用している RDS のストレージは 100 GiB なので、ベースライン性能は 300 IOPS です。

「バーストI/Oクレジット」を使い切ると、ディスク性能はこの 300 IOPS のベースラインまで一気に絞られます。

その「バーストI/Oクレジット」が今どれくらい残っているかを表すのが BurstBalance です。

![](https://static.zenn.studio/user-upload/bc62b1c374cf-20260628.jpg =1000x)

確認した結果、BurstBalance が消費されている場面が多く、障害が発生していた時間帯では 0% になっていました。

バーストI/Oクレジットを使い果たし、ディスク性能が 300 IOPS まで絞られていたということになります。

## 原因のまとめ

ここまでの調査の結果から今回の障害の原因をまとめると、RDS のメモリ不足でディスクへのアクセスが増え続けた結果バーストI/Oクレジットが枯渇し、本来出せていた IOPS すら出せなくなって I/O が詰まっていた、ということになります。

## 対応したこと

原因が「メモリ不足 → ディスク読み込みの増加 → バーストI/Oクレジットの枯渇」という連鎖だったので、その入口と出口の両方に対策を打つことにしました。

RDS を以下のように変更しました。

| 項目 | 変更前 | 変更後 |
| --- | --- | --- |
| インスタンスクラス | db.t4g.micro | db.t4g.small |
| ストレージタイプ | 汎用 SSD (gp2) | 汎用 SSD (gp3) |

### インスタンスクラス

こちらは連鎖の「入口」であるメモリ不足への対策です。

db.t4g.micro の RAM 1GB を、db.t4g.small の 2GB へ倍増させました。

これにより、RDS がメモリ上に保持できるデータ量が増え、ディスクへのアクセス回数が減ることが期待できます。

### ストレージタイプ

こちらは連鎖の「出口」であるバーストI/Oクレジット枯渇への対策です。

gp2 ではボリュームサイズに応じたベースライン性能（今回は 300 IOPS）しかなく、それを超える分はバーストI/Oクレジットを消費して凌ぐ仕組みでした。

一方 gp3 は最初から 3,000 IOPS がベースラインとして使え、「バーストI/Oクレジット」という概念自体がありません。

これにより、「バーストI/Oクレジットを使い切って突然性能が絞られる」という今回のような事象を構造的に防ぐことができます。

## 対応した結果

スペックアップ後、調査時に見ていたメトリクスがどう変化したかを確認しました。

### メモリ

![](https://static.zenn.studio/user-upload/286ffd1273f3-20260628.jpg =1000x)

対応前は 50MB〜150MB ほどしかなかった空きメモリが、RAM の倍増によって大きく増え、十分な余裕が生まれました。

### ReadIOPS / WriteIOPS / TotalIOPS

![](https://static.zenn.studio/user-upload/4c949713bd24-20260628.jpg =1000x)

![](https://static.zenn.studio/user-upload/4ea3dd882035-20260628.jpg =1000x)

![](https://static.zenn.studio/user-upload/5ebcf18898d1-20260628.jpg =1000x)

メモリを増やしたことで、キャッシュにより多くのデータが保持されるようになり、ディスクへのアクセス回数が減りました。

### DiskQueueDepth ( Maximum )

![](https://static.zenn.studio/user-upload/5415f9b1b782-20260628.jpg =1000x)

処理待ちの I/O も落ち着き、行列ができている状態は解消されました。

I/O の詰まりがなくなったことで、アプリの動作もはっきりと改善しました。

## まとめ

RDS において、CPU に余裕があったとしても、ディスク I/O が詰まればアプリは簡単に遅くなります。

そのため、CPU だけでなく FreeableMemory や IOPS、DiskQueueDepth、BurstBalance といった I/O 系のメトリクスもあわせて見ることが大切だと、今回の障害対応を通じて痛感しました。
