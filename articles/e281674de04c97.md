## はじめに

先日、個人開発で運用しているアプリの動作が遅くなる障害が発生しました。

アプリにアクセスしても、なかなか画面が表示されない状況でした。

調査の結果、原因はアプリのコードやアーキテクチャの致命的なミスではなく、単純な RDS のスペック不足でした。

本記事では、今回の障害対応を通じて、改めて RDS について学んだことを整理してみます。

少しでも皆様の参考になりますと幸いです。

## 調査したこと

何が原因でアプリの動作が遅くなっているのかを調査するため、まずは AWS のマネジメントコンソールで CloudWatch のメトリクスを確認しました。

まずはアプリの実行環境である Fargate、次にデータベースの RDS という順で確認しました。

#### CPU

まずは Fargate の CPU 使用率を確認しましたが、一時的に上がっていることはあっても、基本的には低いまま安定していました。

リクエストを捌けずに CPU が振り切れている、という状況ではなさそうです。

#### CPU

#### メモリ

FreeableMemory を確認したところ、空きメモリが常時 100MB〜150MB くらいしかなく、低い時では 50MB を切っていました。

そこで見たのがディスクの読み書き回数です。**ReadIOPS が普段より大きく跳ね上がっていました**。

## 何が起きたか

- ある日の午前10時ごろ、アプリの動作が急に重くなった
- 管理画面を開こうとしても、いつまで経っても表示されない
- アプリのデプロイは直近で行っておらず、コード起因ではなさそう

「最近アクセスが増えてきたとはいえ、急に落ちるほどか……？」と思いつつ、まずは AWS のマネジメントコンソールから RDS のメトリクスを確認することにしました。

## 原因調査：CloudWatch で見た3つのメトリクス

RDS の調査では、CloudWatch のメトリクスがほぼすべてを教えてくれました。今回チェックして "効いた" のは次の3つです。

### 1. FreeableMemory（空きメモリ）

空きメモリが常時 **100MB〜200MB** しかありませんでした。

RDS（MySQL/PostgreSQL）は、空いているメモリをバッファプール（キャッシュ）として使い、よく読むデータをメモリ上に保持することでディスクアクセスを減らしています。空きメモリがほとんどない状態だと、**キャッシュにデータを乗せきれず、毎回ディスクまで読みに行く**ことになります。

### 2. BurstBalance（I/Oクレジット残量）

そして決定打がこれ。**BurstBalance が 0% に張り付いていました。**

これは gp2 ボリュームの「バーストI/Oクレジット」の残量を表すメトリクスです。後述しますが、これが 0 になるとディスクの性能がベースラインまで一気に絞られ、I/O が詰まります。

### 3. Read IOPS / DiskQueueDepth

ディスクへの読み込み回数（Read IOPS）が増え、処理を待っている I/O の数（DiskQueueDepth）も上昇していました。「ディスクに読みに行く回数が増えているのに、ディスク性能は絞られている」という、いちばん苦しい状態です。

### 何が起きていたのか（負の連鎖）

整理すると、こういう連鎖が起きていました。

1. **メモリが足りず**、よく読むデータをキャッシュに保持できない
2. その結果、**ディスクへの読み込み回数が増える**
3. gp2 のベースライン性能（後述の 300 IOPS）を超えるアクセスを、**バーストでなんとか凌いでいた**
4. しかしバーストには上限があり、ついに **I/Oクレジットが枯渇**
5. ディスク性能がベースラインまで絞られ、**処理が一気に詰まった**

つまり「メモリ不足」と「gp2のクレジット枯渇」が合わさって、表面化したわけです。

## 【ここが肝】gp2 のバーストI/Oクレジットとは

今回のいちばんの学びはここでした。

gp2（汎用SSD）には、次のような性能モデルがあります。

- **ベースライン性能**：`3 IOPS × ボリュームサイズ(GiB)`
- ベースラインを超えるアクセスは、**バーストクレジット**を消費して一時的に高性能を出す
- クレジットには上限があり、使い切るとベースライン性能まで絞られる

私のケースはストレージ **100 GiB** だったので、ベースラインは `3 × 100 = 300 IOPS`。普段はこの 300 IOPS で足りていても、メモリ不足でディスク読み込みが増えるとあっという間にベースラインを超え、バーストで凌ぎ続けた結果、クレジットが尽きてしまったというわけです。

ポイントは、**ボリュームが小さいほどベースラインが低く、クレジットも枯渇しやすい**こと。「容量は余ってるからストレージは小さくていいや」という判断が、思わぬ性能問題を呼ぶことがあります。

### gp3 なら何が変わるのか

これを根本から解決してくれるのが gp3 です。

| | gp2 | gp3 |
| --- | --- | --- |
| ベースラインIOPS | 3 IOPS/GiB（容量依存） | **3,000 IOPS（容量と独立）** |
| バーストクレジットの概念 | あり（枯渇する） | **なし** |
| IOPS/スループットの個別設定 | 不可 | **可能** |
| コスト | gp3 より割高なことが多い | ストレージ単価が安め |

gp3 は最初から 3,000 IOPS がベースラインとして使え、**「クレジット枯渇」という概念自体がありません**。容量と性能が切り離されているので、「容量は100GiBのままでIOPSだけ上げる」といった調整もできます。

## 対処：スペックアップ

調査結果を踏まえ、メモリとストレージの両方に手を入れました。

| 項目 | 変更前 | 変更後 |
| --- | --- | --- |
| インスタンスクラス | db.t4g.micro | **db.t4g.small** |
| vCPU | 2 | 2 |
| RAM | 1 GB | **2 GB** |
| ストレージタイプ | 汎用SSD (gp2) | **汎用SSD (gp3)** |
| ストレージ | 100 GiB | 100 GiB |
| プロビジョンドIOPS | 300 IOPS | **3,000 IOPS** |
| スループット | 125 MiBps | 125 MiBps |

### なぜ db.t4g.small か

RAM を 1GB → 2GB に倍増させ、**キャッシュに乗せられるデータ量を増やす**のが狙いです。空きメモリが増えればディスク読み込みそのものが減り、I/O問題の根っこが軽くなります。

### なぜ gp2 → gp3 か

前述のとおり、gp3 にすれば**バーストI/Oクレジットという概念がなくなり、常時 3,000 IOPS が使える**ため、今回のような「クレジット枯渇で突然遅くなる」事象を構造的に防げます。コスト面でもストレージ単価が安く、コスパが良いです。

## コストはどれくらい上がったか

個人開発でいちばん気になるのがここ。スペックアップによる月額の変化を試算しました（1ドル160円換算、Multi-AZのProduction＋Single-AZのStagingの2環境を運用しているケース）。

**スペックアップ前**

- Production (Multi-AZ)：$0.05 × 24h × 31days = $37.2
- Staging (Single-AZ)：$0.025 × 24h × 31days = $18.6
- 合計：$55.8 ≒ **8,928円**

**スペックアップ後**

- Production (Multi-AZ)：$0.101 × 24h × 31days = $75.14
- Staging (Single-AZ)：$0.05 × 24h × 31days = $37.2
- 合計：$112.34 ≒ **17,975円**

**→ 毎月およそ +9,047円のコストアップ**

個人開発にとって月+9,000円は決して小さくありません。とはいえ、「管理画面が開けずサービスが使えない」状態が続くダメージを考えれば、ここはケチるところではないと判断しました。**落ちて信頼を失うより、スペックに払うほうが安い**——インフラのコスト判断の典型だなと改めて感じます。

なお、Staging を常時起動ではなく必要なときだけ起動する、ストレージのIOPSだけ上げてインスタンスは据え置く、といった選択肢もあります。自分のサービスの優先度に合わせて調整するとよいでしょう。

## まとめ

個人開発のRDSが突然重くなったインシデントから得た教訓です。

- **t系インスタンス × 小容量gp2** の組み合わせは、メモリ不足とI/Oクレジット枯渇のリスクがある
- 監視しておくべきメトリクスは **FreeableMemory** と **BurstBalance**
  - 空きメモリが慢性的に少ない／BurstBalanceがじわじわ減っているなら危険信号
- 困ってから慌てる前に、**gp3への移行**を検討しておくと安心
- インフラのスペックアップはコストと天秤にかかるが、「落ちる損失」と比べて判断する

同じように「なんかRDSが急に遅い……」と悩んでいる人の助けになれば幸いです。
